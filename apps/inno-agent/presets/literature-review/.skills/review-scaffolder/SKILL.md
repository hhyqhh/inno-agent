# Review Scaffolder

把检索到的文献组织成一篇有叙事主线的综述,而不是一堆摘要的堆砌。

## 何时触发

用户已经(或正在)收集文献,需要把结果组织成综述结构。典型信号:
- "帮我整理一下这些文献"
- "写一篇关于 X 的文献综述"
- "开题报告的研究现状怎么写"
- "这些论文之间有什么分歧"

## 工作流程

1. **清点已有材料**:列出用户已检索到的文献清单(标题/作者/年份/核心结论)。如果清单为空,先提示用户回到检索环节(引导用 `paper-lookup` 或 `storm-research`),不要凭空编。

2. **提炼研究问题**:从用户的题目推导出 1 个主问题 + 2–4 个子问题。例:
   - 主问题:视觉语言模型(VLM)的空间推理能力现状如何?
   - 子问题:① 空间推理如何定义与分类?② 有哪些评测基准?③ 主流方法分哪几条技术路线?④ 当前瓶颈在哪?

3. **归类文献到子问题**:把每篇文献归到最相关的子问题下。一篇文献可属多个子问题。归类后,每个子问题下应有 2 篇以上文献支撑——不足的标注"文献薄弱,需补检索"。

4. **识别三种关系**(这是综述的核心价值,区别于摘要罗列):
   - **共识**:多篇文献共同支持的结论
   - **分歧**:不同文献给出的对立结论或方法路线之争
   - **空白**:子问题下没有文献覆盖,或文献明确指出尚未解决

5. **产出骨架**:按下面的模板输出,先让用户确认结构再展开正文。

## 综述骨架模板

```markdown
# {综述标题}

## 研究问题
{主问题一段话;列子问题}

## 检索说明
- 检索库:{PubMed / OpenAlex / arXiv / ...}
- 检索策略:{关键词 / 过滤条件 / 时间范围}
- 纳入文献:{N 篇}
- 未覆盖:{哪些库/语言/时间段没查到}

## 主流观点
### 子问题 1:{...}
- 共识:{...}[作者 年份][作者 年份]
- 代表方法:{...}

### 子问题 2:{...}
...

## 分歧与争议
- 分歧点 A:{甲方观点} vs {乙方观点}
  - 甲方:[作者 年份] 认为 ...
  - 乙方:[作者 年份] 认为 ...
- 分歧点 B:...

## 研究空白与展望
1. {空白 1:哪个子问题文献不足}
2. {空白 2:文献明确指出的未解问题}
3. 未来方向:{...}

## 参考文献
[1] 作者. 标题. 期刊/会议, 年份. DOI/URL
[2] ...
```

## 原则

- **骨架先行**:第 5 步产出后必须停下让用户确认,再进入正文撰写。避免一次性把正文写完才发现结构不对。
- **显式标注薄弱处**:子问题下文献不足时,不要用模糊话术掩盖,直接标"文献薄弱"。
- **引用必须可追溯**:每个 `[作者 年份]` 必须能在参考文献列表里找到对应条目。无法对应到真实来源的论断,标注"待查证"。
- **不编造文献**:这是硬约束。没有检索到的论文,绝不为了凑数而杜撰。
