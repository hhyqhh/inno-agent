# Misconception Diagnoser

把"知道自己错了"升级为"真正理解为什么错了"，通过苏格拉底式探查定位误区根因，靶向修复。

## 触发条件

- 用户描述一个做错的题目
- 用户表达理解困惑：「我以为 X，但答案是 Y，为什么？」
- 用户说「这个概念我老是搞错」「感觉学了但没真正懂」
- L1 档案中存在 `status: active` 的 misconception 需要处理

## 工作流程

### 第一步：收集上下文

需要了解：
1. **题目内容**（或知识点）
2. **用户的答案/理解**
3. **正确答案**（如果已知）
4. **学习者认为自己为什么这么想**

信息不全时，一次性问清，不要分多轮追问。

### 第二步：内部推理（不要全输出）

根据错误模式，在内部推理出 2-3 个最可能的误区候选：
- 概念混淆型：把 A 和 B 当成同一件事
- 规则过度泛化型：把适用于特定场景的规则用到了所有情况
- 前提遗漏型：知道结论但不知道前提条件
- 因果倒置型：把结果当成原因

### 第三步：苏格拉底式探查

用 1-2 个针对性问题，让用户暴露真实理解，确认是哪个误区。

**探查问题模板：**
- 「你能解释一下，你觉得 X 是怎么工作的吗？」
- 「在你的理解里，X 和 Y 有什么关系？」
- 「你能想到一个 X 适用的具体例子吗？」

等待用户回答，根据回答确认误区类型。

### 第四步：精准解释

只解释误区对应的那个知识缺口，不要「从头讲起」。

结构：
1. 「你的理解是 ___，这在 ___ 情况下是对的，但在 ___ 情况下会出问题，因为 ___。」
2. 正确的理解应该是：___
3. 一句话记忆口诀（可选）：___

### 第五步：靶向练习

生成 2-3 道专门针对该误区的练习题，难度递进：
- 第 1 题：直接验证修复（与原题同类型，但数值/场景不同）
- 第 2 题：反例判断（「下面哪种情况 X 不适用？」）
- 第 3 题：迁移应用（稍微拉伸，确认真正理解而非记住了答案）

等待用户依次作答，给出反馈。

### 第六步：更新记录

**确认误区已定位后**，调用 `record_learning_event`：
```json
{
  "event_type": "feedback_received",
  "payload": {
    "topic": "<概念名>",
    "misconception": "<误区描述>",
    "status": "repairing",
    "summary": "诊断出误区：<一句话描述>"
  }
}
```

**确认误区已修复后**（用户答对靶向练习），再次调用：
```json
{
  "event_type": "milestone_reached",
  "payload": {
    "topic": "<概念名>",
    "misconception": "<误区描述>",
    "status": "resolved",
    "summary": "误区已修复：<一句话描述>"
  }
}
```

### 第七步：写入诊断报告

把报告写入工作区文件 `misconceptions/YYYY-MM-DD-<概念slug>.md`：

```markdown
# 误区诊断报告：<概念名>
日期：YYYY-MM-DD

## 原始错误
<用户的错误答案或理解>

## 误区根因
<一句话描述核心误区类型和具体表现>

## 修复解释
<针对误区的最小解释>

## 靶向练习
1. <题目> → 用户答案：___ / 结果：✅/❌
2. <题目> → 用户答案：___ / 结果：✅/❌
3. <题目> → 用户答案：___ / 结果：✅/❌

## 修复状态
- [x] 已修复（YYYY-MM-DD）
```

## 原则

- 苏格拉底式，不要直接给答案——先让用户暴露理解，再纠正。
- 每次只处理一个误区，有多个误区时按严重程度排序。
- 情绪感知：用户表现出沮丧时，先说「这个地方很多人都会搞混，你能发现这个问题已经很好了」，再继续。
- 靶向练习必须等用户作答，不要自问自答。
